기술적 의사결정

왜 OpenSearch + Nori인가?
MySQL LIKE 검색으로 시작했지만 "볶다"로 검색해도 "제육볶음"이 나오지 않는 등 형태소 단위 처리가 불가했습니다. 한글 형태소 분석이 필요했고, AWS 관리형으로 운영 부담을 줄일 수 있는 OpenSearch + Nori를 선택했습니다. 장애 시 QueryDSL로 자동 폴백되는 이중 구조로 검색이 완전히 죽지 않도록 설계했습니다.
왜 Resilience4j 3중 보호인가?
AI API는 외부 서비스라 언제든 장애가 날 수 있습니다. CircuitBreaker로 연쇄 장애를 차단하고, Retry로 일시적 오류를 재시도하고, TimeLimiter로 무한 대기를 방지합니다. 이 3가지를 조합해 AI 서비스가 죽어도 나머지 기능은 정상 동작하는 구조를 만들었습니다.
왜 Caffeine + Redis 2계층 캐시인가?
Redis만 쓰면 매 요청마다 네트워크를 타야 합니다. 인기 레시피처럼 자주 조회되는 데이터는 Caffeine으로 인메모리에서 즉시 반환하고, 거기 없을 때만 Redis를 조회합니다. 응답시간이 눈에 띄게 줄었습니다.
왜 Multi-AI 전략인가?
Grok(레시피 추출), Gemini API(영상 분석·이미지 생성), Vertex AI(프롬프트 비학습 보장)를 역할별로 분리하고 5개 모델을 비교·선정했습니다. 단일 모델에 의존하면 장애나 비용 급등 시 서비스가 멈추는 리스크가 있습니다. Multi-AI 구조로 모델 교체나 A/B 테스트가 유연해졌습니다.
왜 MDC 컨텍스트 전파인가?
비동기 스레드에서 로그 추적이 끊기는 문제가 있었습니다. TaskDecorator로 부모 스레드의 MDC 컨텍스트를 자식 스레드에 복사하여 traceId가 비동기 흐름에서도 끊기지 않도록 했습니다. 장애 발생 시 요청 단위 로그 추적이 가능해졌습니다.
왜 yt-dlp 기반 YouTube 파이프라인인가?
YouTube 공식 API는 자막을 가져오는 데 제약이 있습니다. yt-dlp로 자막·설명·댓글을 통합 추출하고, User-Agent 회전 + 쿠키 풀로 차단을 우회합니다. 토큰 사용량을 40% 절감하는 컨텍스트 상한 제어와 다단계 AI 파이프라인을 조합해 레시피 추출 정확도를 높였습니다.
왜 Feature Branch 워크플로우인가?
2인 백엔드에서 main 직접 푸시는 충돌과 장애 위험이 컸습니다. Feature Branch 워크플로우로 기능별 브랜치를 분리하고, feat/bugfix/refactor 접두사 + Conventional Commits(한글) 컨벤션을 적용해 394개 브랜치·811 커밋의 이력을 추적 가능하게 관리했습니다. 역할도 본인(AI·검색·인증·인프라) / 팀원(레시피 CRUD·커뮤니티)으로 명확히 나눠 충돌을 최소화했습니다.
왜 BDD 기반 134개 테스트인가?
서비스 로직이 20+ 레이어로 늘어나면서 한 곳을 고치면 다른 곳이 깨지는 회귀 버그가 반복됐습니다. JUnit5 + AssertJ + BDDMockito로 134개 테스트 케이스를 작성하고, BDD Given/When/Then 구조로 통일해 "무엇을 검증하는지"가 코드만 봐도 드러나도록 했습니다. 배포 전 회귀 검증이 자동화되면서 수정 속도가 빨라졌습니다.

Troubleshooting

장애 대응 & 문제 해결

🔥
OOM으로 컨테이너 강제 재시작
상황 추천 API에서 OutOfMemoryError 발생, ECS 컨테이너 강제 재시작으로 서비스 수 분 중단
원인 GC 로그만으로 어떤 코드가 메모리를 점유하는지 특정 불가 → -XX:+HeapDumpOnOutOfMemoryError 플래그 추가 후 재현 시 힙 덤프 수집 → Eclipse MAT 분석 → 추천 로직에서 N개 레시피 × M개 재료의 카르테시안 곱이 수십만 개 객체를 메모리에 적재. 원인 파악까지 이틀 소요.
해결 후보 50개 사전 제한 + 4단계 스코어링(요리 궁합 → 태그 → 재료 → 평점)으로 정확도를 유지하며 메모리 안정화. 이후 Grafana 모니터링으로 재발 추적.
Before
// 전체 레시피 ID를 메모리에 로드
List<Long> allIds =
  recipeRepository.findAllPublicRecipeIds();
Set<Long> uniqueIds = new HashSet<>(candidateIds);
uniqueIds.addAll(allIds); // 수십만 개 객체 생성
After
// 후보 50개로 사전 제한
int candidatePoolSize = 50;
List<Long> selectedIds = candidateIds.stream()
  .filter(id -> !id.equals(currentRecipeId))
  .limit(candidatePoolSize) // 상한 고정
  .collect(Collectors.toList());
✓ OOM 재발 0건 · 힙 사용량 60~70% 안정
⚡
rollback-only 마킹으로 검색 API 전체 마비
상황 검색 API에서 원인 불명의 500 에러 간헐적 발생
원인 일반 에러 로그로는 원인 불명 → org.springframework.transaction 로그 레벨을 DEBUG로 올려 전파 흐름 추적 → OpenSearch 인덱싱 실패가 상위 @Transactional의 트랜잭션을 rollback-only로 마킹 → 커밋 시 UnexpectedRollbackException 발생하는 것 확인.
해결 검색 서비스에 noRollbackFor = CustomException.class 추가해 예외 발생 시 rollback-only 마킹 방지. 인덱싱 실패 건은 별도 테이블에 기록 후 매일 04:00 배치로 자동 복구.
Before
@Transactional(readOnly = true)
public Page<RecipeSimpleDto> searchRecipes(
  RecipeSearchCondition condition,
  Pageable pageable, Long userId)
After
@Transactional(readOnly = true,
  noRollbackFor = CustomException.class)
public Page<RecipeSimpleDto> searchRecipes(
  RecipeSearchCondition condition,
  Pageable pageable, Long userId)
✓ 검색 API 500 에러 0건 · rollback-only 마킹 방지로 읽기 트랜잭션 안정화
🎬
YouTube 자막 추출 서버 IP 차단
상황 yt-dlp 기반 자막 추출이 서버 IP 차단으로 동작 불가
해결 2단계 폴백: 1차(No Cookie + Proxy) → 2차(With Cookie + No Proxy). User-Agent 회전 + 쿠키 풀 적용으로 IP 차단 우회.
✓ 차단 이후에도 자막 추출 기능 무중단 유지
